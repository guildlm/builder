FINDING — gate-chain re-mining over the archived _fail-* corpus (2026-07-24)
==============================================================================

METHOD (model-free): the archived red projects in generated/_fail-* are failures the
builder gave up on WHEN THEY WERE MADE. The deterministic gate chain has grown to 30+
gates since. So re-run the CURRENT chain to a fixpoint over each old failure and see what
it now fixes vs what it still cannot — "whatever it still cannot fix is the backlog"
(tests/test_three_small_gates.py docstring). No model, only go vet + _run_deterministic_gates.

RESULT over 18 _fail-* archives:
  - 15/18 are now GREEN under the current chain. The gates added since these failed
    (self-qualifier, negated-comparison, slice-equal, external-test-package, arity,
    struct-literal-key, ... ) retroactively repair them. This is a strong, concrete
    validation of the whole gate campaign: real historical failures, fixed deterministically.
  - 3/18 remain RED, all with the SAME error:
        vet: ./shortener_test.go:58:36: not enough arguments in call to doReq
    doReq(t, h, method, path, token, body) takes 6 params; the GET-request calls omit the
    trailing `body` (5 args). One recurring mechanical shape.

WHY THIS ONE IS NOT ADDED AS A GATE (the honest call):
  A gate for "not enough arguments" would have to append a value for the missing parameter.
  That is a GUESS, not a single meaning — the two properties the gate philosophy separates
  ("exactly one meaning ... is what separates them from a guess"). Worse, a wrong value
  compiles but fails an ASSERTION, which reads as a model logic bug rather than a mechanical
  one — so a value-guessing gate can actively MISLEAD the fix loop (do harm). Positionally
  the fix is "append the last param's zero value", and for this case ("" for a GET body) it
  is even semantically right, but the gate cannot know that in general. So this gap belongs
  to the MODEL (or a risk-accepting heuristic), not to the deterministic chain.

TAKEAWAY: the deterministic gate chain is in good shape — it clears 15/18 of the archived
backlog. The remaining recurring failure is an arity/value mismatch that is deliberately
left to the model. This is a NEGATIVE result on "add another gate" and a POSITIVE result on
"is the chain effective". The one clean gate this session (external-test-package) came from
a live demo, not this corpus — the corpus's remaining tail is model-shaped, not gate-shaped.
