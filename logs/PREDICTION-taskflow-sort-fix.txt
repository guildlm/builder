=== CONTEXT: the sort family needs a TOOL change, not just a spec edit ===
The 3 sort holes (taskflow/usersapi/taskapipro) share a verification problem: the teeth
mutation DROPPED the sort, and a dropped sort yields Go map-iteration order (random), so
the catch is probabilistic (the tasks-api ~20% lesson). A real order-asserting test DOES
defend the invariant — deterministically against a REVERSED sort. So the principled fix
is: (a) add an order-asserting test to the spec, AND (b) switch the teeth entry from DROP
to REVERSE (ascending->descending), which an ascending assertion catches every run. This
mirrors the taskapi POSITIVE CONTROL, which already uses reverse. Done here for taskflow.
